# manual-sync

## 무엇을 하는가

기존 매뉴얼의 **부분 패치**를 안전하게 수행한다. `manual-writer-agent` 가 본 스킬의 원칙을 따라 본문을 수정한다.

## 작성 표준 (필수 참조)

본문 작성·수정 전 **`docs/매뉴얼_작성_표준_가이드.md`** 를 읽고 다음 원칙을 적용한다 (IEC/IEEE 82079-1 + ISO/IEC/IEEE 26514 기반):

1. **정보 3유형 분류** — 개념(무엇/왜) · 지침(어떻게, 번호 목록) · 참조(용어집·트러블슈팅·표)를 섞지 않는다.
2. **대상 독자별 구성** — 신규 사용자/테스터/PM/관리자 등 독자가 다른 내용은 섹션을 분리한다. 머리말의 "이 매뉴얼을 읽는 방법" 표를 유지·갱신한다.
3. **절차는 목록으로** — 단락 서술 대신 번호/불릿. 한 문장 = 한 아이디어.
4. **경고·팁 표기 일관성** — ⚠️ 주의 / 💡 팁 / `> **날짜 변경**` 변경 메모. 다른 이모지·표기 발명 금지.
5. **개정 이력 갱신** — 변경 시 문서 말미 "변경 이력" 표에 한 행 추가.

대상 파일:
1. **메인(한국어)**: `docs/manual/new/USER_MANUAL.md` (18 섹션) — 한 페이지 완전판
2. **영문판**: `docs/manual/new/USER_MANUAL_EN.md` — 한국어판과 섹션·소제목 1:1 대응 (이미지는 `images_en/`)
3. **챕터별**: `docs/manual/*.md` (14개) — 항목별 분리본

같은 변경이 여러 곳에 영향을 주면 전부 갱신한다. **특히 한국어판 본문을 수정하면 영문판의 같은 섹션도 반드시 같이 수정한다** — 영문판만 뒤처지는 drift 가 가장 흔한 사고다. 영문판 수정 시: 간결·명령형 영어(ISO/IEC/IEEE 26514 톤), 이미지 경로는 `images_en/`, 내부 상호 참조는 앵커 링크 대신 "see Section N-N" 일반 텍스트, UI 라벨은 실제 영어 UI 문구와 1:1(예: "Add Sub Folder").

동기화 검증: `python3 -c` 로 양쪽의 `### N-N.` 소제목 목록을 추출해 diff 하면 누락 섹션을 기계적으로 잡을 수 있다.

## 핵심 원칙

| 원칙 | 적용 |
|------|------|
| **변경분만** | 전면 재작성·재배열 금지. 패치는 항상 부분적 |
| **코드 기준** | 매뉴얼과 코드가 다르면 코드를 진실로 봄. 컴포넌트 파일·라우트 정의(`src/main/frontend/src/App.jsx`)·data-testid 확인 |
| **백틱 일관성** | 경로(`/projects`, `/dashboard`)는 백틱으로 — manual_capture.py 감사 매처가 백틱 안 경로만 추출 |
| **이미지 참조 보존** | `![한국어 캡션](images/NN_slug.png)` 패턴 유지. 슬러그 변경 시 manual_capture.py STEPS 도 같이 변경 |
| **톤** | 한국어 존댓말, 친근하지만 간결. 명령형/지시형 (`~합니다`, `~하세요`) |
| **추측 금지** | 코드에서 확인 안 되는 기능 적지 않음 |

## 패치 패턴 — 누락 경로 → 신규 섹션

manual-capture 의 감사 리포트가 `/projects/:id/testcases` 누락이라고 보고했고, 실제 매뉴얼 §4 가 화면 흐름만 적고 URL 백틱이 빠져 있으면:

**Before:**
```markdown
### 4-1. 화면 구성

프로젝트 대시보드에서 **테스트케이스** 탭을 클릭하면...
```

**After:**
```markdown
### 4-1. 화면 구성

프로젝트 대시보드에서 **테스트케이스** 탭을 클릭하면 `/projects/{projectId}/testcases` 화면이 표시됩니다.
```

→ URL 한 줄 추가로 감사 통과, 본문 흐름 변경 0.

## 패치 패턴 — 코드 변경 반영

PR 커밋 메시지나 사용자 보고에서 "UI 변경" 단서가 있으면:

1. 변경된 컴포넌트 파일 확인 (예: `TestCaseSpreadsheet.jsx` 의 한글 30자 워드랩)
2. 매뉴얼에서 그 화면을 다루는 섹션 찾기 (§4-1 또는 §4-4)
3. 변경된 동작에 한 줄 보강:
   ```markdown
   > **2026-05-28 변경**: 셀 내용이 한글 30자(약 30em)를 넘으면 자동으로
   > 워드랩되어 다음 줄로 이어집니다. 컬럼 폭이 더 이상 가로로 늘어나지 않습니다.
   ```
4. 캡처 갱신이 필요하면 manual-capture 호출 안내

## 패치 패턴 — 라우트 제거

코드에서 라우트가 사라졌고, 매뉴얼에 백틱으로 남아 있으면:
1. 본문 검색: `grep -n "/removed-route" docs/manual/new/USER_MANUAL.md`
2. 해당 섹션 제거 또는 "기존 X 기능은 Y 로 대체됨" 메모로 변환
3. 이미지 파일은 즉시 삭제하지 말고 `docs/manual/new/images/_archive/` 로 이동 (롤백 가능성)

## 패치 패턴 — 챕터 매뉴얼 ↔ 통합 매뉴얼 동기화

`docs/manual/user_management_manual.md` 같은 챕터 매뉴얼이 별도로 존재한다. 통합본 §17 의 사용자 관리 섹션과 같은 정보를 다른 깊이로 다룬다:
- 통합본: 사용자가 처음 둘러볼 수준의 개요
- 챕터본: 운영자가 필요할 때 깊이 들여다보는 레퍼런스

한쪽이 갱신되면 다른 쪽도 같이 보강한다. 정보 중복은 허용 (각각 다른 독자), 하지만 모순은 허용 안 함.

이전 사례: `bdf71b6a docs(manual): user_management_manual.md 실제 코드와 동기화` — 챕터본만 갱신, 통합본은 그대로 → 다음 갱신 시 통합본 §17 도 같이 확인할 것.

## 검증 절차 (패치 후 필수)

본문 패치가 끝나면 **커밋 전에** 두 개의 검증 에이전트를 병렬로 띄워 통과해야 한다.

### 1단계 — 누락 비교분석 (Explore agent)

코드 ↔ 매뉴얼 전수 대조. 프롬프트에 반드시 포함할 것:

- `src/main/frontend/src/App.jsx` 의 모든 라우트 + `/*` 캐치올 내부 뷰 vs 매뉴얼의 백틱 경로
- 헤더/관리 메뉴/프로필 탭 등 메뉴 항목 목록 vs 매뉴얼 해당 섹션
- 매뉴얼의 구체적 수치·제한 주장(최대 개수, 파일 크기, 만료 기간, enum 값) → 코드 근거 파일:라인 요구
- 최근 릴리즈 노트(`docs/release_note/`)의 사용자 가시 변경 vs 매뉴얼 반영 여부

출력: [누락/불일치/일치] 판정 표. **불일치·누락은 패치 후 재검증.**

### 2단계 — 문서 품질·용어 검증 (문서 전문가 agent)

`docs/매뉴얼_작성_표준_가이드.md` 의 체크리스트를 기준으로 다음을 검수:

- **사용자 친화 용어** — 개발자 용어(엔드포인트·컴포넌트·prop·DTO 등)가 본문에 새어 들어갔는지. 일반 사용자가 모르는 용어는 풀어쓰거나 §18 용어집에 등재됐는지
- **문장 품질** — 한 문장 한 아이디어, 절차의 목록화, 존댓말 톤 일관성
- **표기 일관성** — ⚠️/💡/변경 메모 규칙, 경로 백틱, 이미지 alt 텍스트 한국어
- **구조** — 정보 3유형이 섞이지 않았는지, 독자별 안내 표와 목차가 본문과 일치하는지

출력: 심각도(차단/권고/참고)별 지적 목록. **차단 항목은 수정 후 종료 보고.**

두 에이전트 결과는 종료 보고의 "검증 결과" 절에 요약 기록한다.

## 안티패턴 (피할 것)

- ❌ 전면 재작성 — 톤 일관성·git diff 노이즈
- ❌ 섹션 번호 재배열 — 이미지 파일명 ↔ 섹션 번호 매핑 깨짐
- ❌ 코드에 없는 기능 설명 — 잘못된 정보 전파
- ❌ 백틱 없는 경로 표기 — 감사 매처에서 인식 못 함
- ❌ 영문 음역 ("아이콘", "버튼", "메뉴" 는 한국어로 적되 영어 용어 "modal", "dialog" 가 더 정확하면 백틱)
- ❌ 이미지 alt 텍스트 누락/영어
