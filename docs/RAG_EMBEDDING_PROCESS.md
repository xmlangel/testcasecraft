# RAG 문서 임베딩 프로세스

이 문서는 본 프로젝트의 RAG(Retrieval-Augmented Generation) 시스템에서 문서가 어떻게 임베딩되는지에 대한 과정을 설명합니다.

## 1. 시스템 아키텍처 개요

RAG 시스템은 3-Tier 아키텍처로 구성되어 있습니다.

```
React Frontend -> Spring Boot Backend -> FastAPI RAG Service
```

- **Frontend**: 사용자가 문서를 업로드하고 검색을 요청하는 인터페이스입니다.
- **Spring Boot Backend**: Frontend와 FastAPI 서비스 간의 통신을 중계하고 비즈니스 로직을 처리합니다.
- **FastAPI RAG Service**: 문서 처리, 임베딩 생성, 벡터 검색 등 핵심 RAG 기능을 수행합니다.

임베딩 프로세스는 주로 **FastAPI RAG Service**에서 이루어집니다.

## 2. 임베딩 프로세스 흐름

문서 업로드부터 임베딩 벡터가 데이터베이스에 저장되기까지의 과정은 다음과 같습니다.

**1단계: 문서 업로드 (Frontend -> Backend)**
- 사용자가 React 기반의 프론트엔드에서 문서를 업로드합니다.
- 문서는 Spring Boot 백엔드로 전송됩니다.

**2단계: 문서 저장 및 처리 요청 (Backend -> FastAPI)**
- Spring Boot 백엔드는 받은 문서를 FastAPI RAG 서비스로 전달합니다.
- FastAPI 서비스는 문서를 `MinIO`와 같은 S3 호환 오브젝트 스토리지에 저장합니다.

**3단계: 문서 파싱 및 청킹 (FastAPI)**
- FastAPI 서비스 내의 `upstage_service.py` 또는 유사한 파서가 문서를 텍스트로 변환합니다.
- 변환된 텍스트는 의미 있는 단위의 작은 조각(Chunk)으로 분할됩니다. 이 과정은 문서의 구조와 내용을 고려하여 이루어집니다.
- 각 청크는 원본 문서와의 관계 정보와 함께 PostgreSQL 데이터베이스에 저장됩니다.

**4단계: 임베딩 생성 (FastAPI -> SentenceTransformer)**

이 단계는 문서의 텍스트 조각(Chunk)을 의미를 담은 숫자 벡터(Vector)로 변환하는 핵심 과정입니다.

1.  **프로세스 트리거**:
    *   문서가 성공적으로 파싱되고 청크로 분할된 후, 임베딩 생성 프로세스가 시작됩니다.
    *   보통 문서 업로드 직후 자동으로 실행되거나, 사용자가 UI에서 수동으로 분석을 시작할 때 트리거됩니다.

2.  **임베딩 대상 청크 조회**:
    *   `embedding_service.py`는 PostgreSQL 데이터베이스를 쿼리하여 아직 임베딩이 생성되지 않은 텍스트 청크를 조회합니다.
    *   API 호출 효율성과 데이터베이스 부하를 고려하여, 한 번에 여러 개의 청크를 '배치(Batch)' 단위로 가져옵니다.

3.  **모델 로드 및 데이터 준비**:
    *   `embedding_service.py`는 `SentenceTransformer` 라이브러리를 사용하여 임베딩 모델을 로드합니다. 기본 모델은 `"paraphrase-multilingual-mpnet-base-v2"`입니다.
    *   조회된 각 텍스트 청크에서 순수 텍스트 내용을 추출하여 모델 입력 형식에 맞게 준비합니다.

4.  **SentenceTransformer 모델 호출**:
    *   `embedding_service.py`는 로드된 `SentenceTransformer` 모델의 `encode` 메서드를 호출하여 텍스트 청크에 대한 임베딩을 생성합니다.
    *   이 모델은 텍스트를 직접 입력받아 벡터를 반환합니다.

5.  **임베딩 벡터 수신**:
    *   `encode` 메서드는 각 텍스트 청크에 대한 임베딩 벡터(실수 배열)를 반환합니다.
    *   `"paraphrase-multilingual-mpnet-base-v2"` 모델의 경우, 768차원의 벡터를 생성합니다.

    *   **예시 임베딩 벡터**:
        ```
        [
          -0.006929283495992422,
          -0.005336422473192215,
          ... (766개의 추가 숫자)
        ]
        ```

6.  **데이터베이스에 벡터 저장**:
    *   `embedding_service.py`는 생성된 임베딩 벡터를 해당 텍스트 청크의 ID와 매칭시킵니다.
    *   PostgreSQL 데이터베이스에 `UPDATE` 쿼리를 실행하여, 특정 청크 ID를 가진 레코드의 벡터 컬럼(`vector` 등)에 수신된 임베딩 벡터를 저장합니다.
    *   `pgvector` 확장은 이 숫자 배열을 효율적인 바이너리 형식으로 저장하여 나중에 유사도 검색을 빠르게 수행할 수 있도록 합니다.

7.  **오류 처리 및 재시도**:
    *   모델 로드 또는 임베딩 생성 중 발생할 수 있는 오류에 대한 처리 로직이 포함되어 있습니다.

8.  **프로세스 완료**:
    *   한 배치의 모든 청크에 대한 임베딩이 성공적으로 데이터베이스에 저장되면, 해당 배치의 작업이 완료됩니다.
    *   아직 처리되지 않은 청크가 남아있다면, 다음 배치를 가져와 위 과정을 반복합니다.

**5단계: 벡터 저장 (FastAPI -> PostgreSQL with pgvector)**
- 생성된 벡터 임베딩은 해당 텍스트 청크의 ID와 함께 PostgreSQL 데이터베이스에 저장됩니다.
- 데이터베이스에는 `pgvector` 확장이 설치되어 있어, 고차원 벡터 데이터를 효율적으로 저장하고 검색할 수 있습니다.

## 3. 주요 기술 요소

- **FastAPI**: RAG의 핵심 로직을 처리하는 파이썬 기반 웹 프레임워크입니다.
- **`embedding_service.py`**: FastAPI 서비스 내에서 텍스트 청크를 받아 임베딩 모델을 호출하는 역할을 담당합니다.
- **SentenceTransformer**: 텍스트를 벡터로 변환하는 임베딩 모델을 제공합니다. (기본 모델: `paraphrase-multilingual-mpnet-base-v2`)
- **PostgreSQL with `pgvector`**: 원본 텍스트 청크와 그에 해당하는 벡터 임베딩을 함께 저장하는 데이터베이스입니다. `pgvector`는 벡터 간의 유사도 검색(코사인 유사도 등)을 빠르게 수행할 수 있도록 지원합니다.
- **MinIO**: 업로드된 원본 문서를 저장하는 S3 호환 오브젝트 스토리지입니다.

**5단계: 벡터 저장 (FastAPI -> PostgreSQL with pgvector)**
- 생성된 벡터 임베딩은 해당 텍스트 청크의 ID와 함께 PostgreSQL 데이터베이스에 저장됩니다.
- 데이터베이스에는 `pgvector` 확장이 설치되어 있어, 고차원 벡터 데이터를 효율적으로 저장하고 검색할 수 있습니다.

## 3. 주요 기술 요소

- **FastAPI**: RAG의 핵심 로직을 처리하는 파이썬 기반 웹 프레임워크입니다.
- **`embedding_service.py`**: FastAPI 서비스 내에서 텍스트 청크를 받아 임베딩 모델 API를 호출하는 역할을 담당합니다.
- **OpenAI**: 텍스트를 벡터로 변환하는 임베딩 모델을 제공합니다. (예: `text-embedding-ada-002`)
- **PostgreSQL with `pgvector`**: 원본 텍스트 청크와 그에 해당하는 벡터 임베딩을 함께 저장하는 데이터베이스입니다. `pgvector`는 벡터 간의 유사도 검색(코사인 유사도 등)을 빠르게 수행할 수 있도록 지원합니다.
- **MinIO**: 업로드된 원본 문서를 저장하는 S3 호환 오브젝트 스토리지입니다.

이러한 과정을 통해 업로드된 문서는 검색 가능한 벡터 데이터로 변환되며, 사용자가 질문을 입력했을 때 가장 관련성 높은 문서 조각을 빠르게 찾아내어 답변 생성에 활용할 수 있게 됩니다.
