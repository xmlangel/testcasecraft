## 1. 회원가입과 로그인

### 1-1. 로그인 화면

`http://localhost:8080/` 또는 `/login` 으로 접속하면 로그인 화면이 표시됩니다.

![로그인 화면](images/01_login_empty.png)

기존 계정이 있다면 **아이디·비밀번호**를 입력하고 **[로그인]** 버튼을 클릭합니다. 처음 사용한다면 하단의 **[회원가입]** 버튼을 클릭해서 계정을 만듭니다.

### 1-2. 회원가입 폼

회원가입 모드로 전환되면 다음 5개 필드가 표시됩니다.

![회원가입 빈 폼](images/02_signup_empty.png)

| 필드 | 설명 |
|---|---|
| 아이디 | 로그인용 식별자 (영문/숫자 권장) |
| 비밀번호 | 8자 이상 권장 |
| 비밀번호 확인 | 위와 동일하게 다시 입력 |
| 이름 | 화면에 표시될 사용자 이름 |
| 이메일 | 인증·알림 발송용 이메일 주소 |

모든 필드를 채운 모습입니다.

![회원가입 입력 완료](images/03_signup_filled.png)

**[회원가입]** 버튼을 누르면 백엔드에 등록되고, 자동으로 로그인 화면으로 돌아옵니다(성공 안내 표시).

![회원가입 후 로그인 화면](images/04_signup_complete.png)

> ⚠️ **에러 메시지가 보이면** — `Failed to fetch`처럼 네트워크 오류가 표시되는 경우 백엔드/프록시 설정 문제입니다. 도커 환경에서는 `REACT_APP_API_BASE_URL`이 실제 접속 URL과 일치해야 합니다 ([§8](#8-부록--jira-연동-환경-변수) 참고).

### 1-3. 로그인

방금 만든 계정으로 로그인합니다.

![로그인 입력](images/05_login_filled.png)

성공하면 **프로젝트 관리** 화면(`/projects`)으로 이동합니다.

## 2. 프로젝트 만들기

신규 사용자는 참여 중인 프로젝트가 없으므로 빈 화면이 보입니다.

![프로젝트 관리 빈 상태](images/10_projects_empty.png)

우상단의 **[+ 새 프로젝트 생성]** 또는 가운데의 **[+ 프로젝트 생성]** 버튼을 클릭하면 다이얼로그가 열립니다.

### 2-1. 프로젝트 생성 다이얼로그

![프로젝트 생성 다이얼로그](images/11_project_create_dialog.png)

다이얼로그에서 다음을 입력합니다.

| 항목 | 예시 | 비고 |
|---|---|---|
| 프로젝트 이름 | `샘플 프로젝트` | 화면에 표시되는 이름 |
| 프로젝트 코드 | `SMP` | 테스트케이스 식별 ID 접두어 (예: `SMP-001`) |

![입력 완료](images/12_project_create_filled.png)

**[생성]** 버튼을 누르면 프로젝트가 만들어지고 카드 형태로 노출됩니다.

### 2-2. 생성된 프로젝트 카드

![프로젝트 생성 완료](images/13_project_created.png)

- 상단 **[독립 프로젝트] / [전체 프로젝트]** 탭으로 범위 전환
- 카드에는 **케이스 수 / 멤버 수 / 자동화 수** 가 작게 표시
- **[프로젝트 열기]** 버튼으로 진입

## 3. 화면 구성과 헤더

프로젝트에 진입하면 상단 헤더 + 좌측 트리 + 우측 본문의 3분할 레이아웃이 표시됩니다.

![프로젝트 대시보드](images/20_project_overview.png)

### 3-1. 헤더 구성

| 영역 | 설명 |
|---|---|
| 좌측 로고 `TESTCASE CRAFT` | 클릭 시 프로젝트 목록(`/projects`)으로 |
| 브레드크럼 `프로젝트 / 샘플 프로젝트 / 대시보드` | 현재 위치 표시 |
| 탭바 `대시보드 / 테스트케이스 / 테스트 플랜 / 테스트실행 / 테스트결과 / 자동화 테스트 / RAG 문서 / 탐색 세션` | 프로젝트 내 영역 전환 |
| 우상단 `프로젝트 선택` | 다른 프로젝트로 빠르게 이동 |
| `⚠ JIRA` 배지 | JIRA 미설정 시 노란 경고 — 클릭하면 설정 안내 |
| 🌓 (테마 아이콘) | 다크/라이트 모드 토글 |
| 사용자 아바타 (이니셜) | 클릭 시 프로필/로그아웃 메뉴 |

### 3-2. JIRA 패널

헤더의 **JIRA** 배지를 클릭하면 연동 설정 화면으로 이동합니다.

![JIRA 패널](images/45_jira_panel.png)

## 4. 테스트 케이스 작성

상단 **[테스트케이스]** 탭으로 진입합니다.

![테스트케이스 페이지 — 빈 상태](images/21_testcase_page.png)

### 4-1. 화면 구성

| 영역 | 설명 |
|---|---|
| 좌측 패널 (≈280px) | **테스트 케이스 트리** — 폴더와 케이스의 계층 구조 |
| 좌측 상단 도구 행 | ☑ 전체체크 / 📁 폴더수 / 📄 케이스수 / 🔄 새로고침 / ↕ 정렬 / `<` 패널 접기 |
| 우측 본문 | **테스트케이스 스프레드시트** — 한 번에 여러 케이스를 입력하는 표 형태 편집기 |
| 본문 상단 도구 | `+ 행 추가`, `위에 추가`, `아래에 추가`, `폴더 추가`, `삭제`, `검증`, `Import/Export`, `Export`, `톱니바퀴`, `전체화면`, `일괄 저장` |

### 4-2. 입력 모드 선택

본문 상단의 **[입력 모드 선택 펼치기]** 를 클릭하면 입력 방식을 전환할 수 있습니다.

![입력 모드 옵션](images/44_input_mode_open.png)

- **스프레드시트 모드** — 표로 여러 케이스 일괄 입력 (기본)
- **폼 모드** — 한 건씩 자세히 입력
- **고급 스프레드시트** — 외부 시트 패턴

### 4-3. 행 추가 다이얼로그

스프레드시트 상단의 **[+ 행 추가]** 를 누르면 추가할 행 개수를 묻는 다이얼로그가 뜹니다.

![행 추가](images/22_tree_add_menu.png)

기본 5개. 1–100개까지 한 번에 추가 가능. **[추가]** 를 누르면 빈 행이 생기고 ID/이름/설명/사전조건/스텝 등을 입력합니다. 마지막으로 우상단 **[💾 일괄 저장]** 으로 한 번에 저장합니다.

### 4-4. 폴더와 케이스 트리 구조

testcasecraft는 폴더와 케이스를 **하나의 트리**로 관리합니다.

- **폴더** (`📁`) — 자식을 가질 수 있는 분류 노드
- **케이스** (`📄`) — 실제 테스트 시나리오
- **시스템 폴더** — `[SYSTEM] 기본 폴더 - 삭제불가` 설명을 가진 보호 폴더 (이동·삭제 불가)

좌측 트리 헤더의 **➕** 아이콘(권한이 있을 때 노출) 또는 트리에서 **우클릭** → 컨텍스트 메뉴로 폴더/케이스를 추가할 수 있습니다.

## 5. 테스트 케이스 트리 — 드래그앤드롭 재구성

> 본 기능은 v1.0.77+에서 제공됩니다. 자세한 설계는 [`docs/plan/TREE_DND_REORGANIZE_PLAN.md`](../../plan/TREE_DND_REORGANIZE_PLAN.md), 사용자 가이드는 [`docs/guide/TREE_DND_USER_GUIDE.md`](../../guide/TREE_DND_USER_GUIDE.md) 참고.

### 5-1. 동작 요약

기존의 **우클릭 메뉴**(추가/이름변경/삭제)와 **순서 편집 모드**(↑/↓ 버튼)는 그대로 사용 가능하며, 추가로 다음과 같이 **마우스 드래그**로 위치를 바꿀 수 있습니다.

| 하고 싶은 것 | 동작 |
|---|---|
| 케이스를 다른 폴더로 옮기기 | 케이스 행 좌측 **`⋮⋮`(드래그 핸들)** 을 잡고 끌어 폴더 위에 놓기 |
| 폴더 통째로 옮기기 | 폴더 행의 드래그 핸들을 잡고 다른 폴더 위에 놓기 |
| 같은 부모 내 순서 바꾸기 | 노드 사이의 **가는 갭(상단 6px / 하단 6px)** 에 놓기 |
| 루트로 이동 | 트리 최상단 빈 영역에 놓기 |

드롭 직전 어디에 들어갈지 **연한 가이드 라인** 으로 미리 보입니다.

### 5-2. 다중 선택 이동

여러 항목을 한 번에 옮길 수 있습니다.

1. **체크박스** 또는 `Cmd-click`(Mac) / `Ctrl-click`(Win)로 여러 노드 선택
2. 선택된 묶음 중 **하나의 드래그 핸들** 을 잡고 끌기 → 묶음 전체가 함께 이동
3. 백엔드는 단일 트랜잭션의 **`POST /api/testcases/move-batch`** 한 번만 호출되며, 하나라도 실패하면 전체 롤백

### 5-3. 자동 차단되는 이동

다음은 드래그 중 가이드가 **빨간색** 으로 바뀌며 자동 차단됩니다.

- 자기 자신/하위(후손) 폴더에 놓기 (순환 방지)
- 다른 프로젝트 폴더에 놓기
- 케이스 **아래에** 다른 노드를 놓기 (케이스는 자식 불가)
- 시스템 기본 폴더를 이동

거부 시 토스트로 사유를 알려주며, 트리 상태는 즉시 원래대로 복원됩니다.

### 5-4. 감사 로그

모든 성공한 이동은 백엔드 `tc_move_audit_log` 테이블에 기록됩니다 — 누가, 언제, 어디서 어디로 옮겼는지 추적 가능. 관리자는 DB 또는 향후 관리 화면에서 조회할 수 있습니다.

## 6. 대시보드와 통계

프로젝트 상단 **[대시보드]** 탭은 그 프로젝트의 핵심 지표를 한 화면에 정리합니다.

![대시보드](images/46_dashboard.png)

| 카드 | 설명 |
|---|---|
| 프로젝트 요약 | 케이스 수 / 참여 인원 |
| 최근 테스트 결과 | 성공/실패/미실행/건너뜀/차단됨 비율 |
| 테스트 결과 추이 | 최근 N일 라인 차트 |
| 열린 테스트 실행 결과 | 진행 중인 실행 요약 |
| 담당자별 결과 | 멤버별 케이스 진행 상황 |
| 테스트 플랜별 결과 | 플랜 단위 통계 |

상단 **[새로고침]** 으로 최신화. **[최근 15일 ▾]** 드롭다운으로 기간 변경.

## 7. 테스트 플랜

상단 **[테스트 플랜]** 탭에서 한 묶음의 테스트케이스를 실행 단위로 묶은 **테스트 플랜**을 관리합니다.

![테스트 플랜 목록 — 비어있음](images/51_testplans.png)

- 우상단 **[+ 새 플랜]** 버튼으로 신규 플랜 생성 (이름, 설명, 포함할 케이스 선택)
- 플랜 목록에서 카드/행 클릭 → 상세 진입
- 플랜에 묶인 케이스는 "테스트 실행"에서 한 번에 실행 가능

> 신규 프로젝트라 플랜이 비어 있는 상태입니다. 플랜을 만들면 카드로 보입니다.

## 8. 테스트 실행

상단 **[테스트실행]** 탭은 테스트 플랜에 묶인 케이스들의 실행 인스턴스를 관리합니다.

![테스트 실행 목록 — 비어있음](images/52_executions.png)

- 검색창으로 제목 필터링
- 행 클릭 → 그 실행 내부로 진입, 각 케이스의 결과(통과/실패/건너뜀/차단)를 기록
- "테스트 결과" 탭은 종료된 실행의 통계를 종합 조회

수동 실행은 다음 흐름입니다.

1. 테스트 플랜에서 **[실행 시작]** 클릭
2. 케이스 하나씩 진행 — 단계별 expected/actual 입력 + Pass/Fail 마킹
3. 첨부 파일(스크린샷, 로그 등) 업로드 가능
4. 완료 시 결과는 자동으로 **테스트 결과** 와 **대시보드 통계**에 반영

## 10. 자동화 테스트

상단 **[자동화 테스트]** 탭은 외부 CI(JUnit/Playwright/Pytest 등)에서 만든 결과 XML/JSON을 업로드해 testcasecraft에서 추적하는 기능입니다.

![자동화 테스트](images/54b_automation.png)

- **JUnit XML 업로드** — `*.xml` (최대 100MB) 업로드

## 16. 부록 — JIRA 연동, 환경 변수

### 8-1. JIRA 연동

헤더의 **`⚠ JIRA`** 배지는 JIRA 설정이 비어 있다는 경고입니다. 클릭하면 JIRA 연동 설정 화면이 열립니다.

![JIRA 설정 패널](images/45_jira_panel.png)

- 서버 URL, 사용자 이메일, API 토큰 입력
- 매핑 필드(우선순위, 담당자 등) 설정
- 저장 후 경고 배지가 사라집니다

### 8-2. 도커로 띄울 때 권장 환경 변수

`docker-compose-build/.env` 또는 셸 환경에서 다음을 설정하세요. 잘못 설정하면 `Failed to fetch` 에러가 발생합니다.

```bash
PROTOCOL=http
DOMAIN=localhost
HTTP_PORT=8080
HTTPS_PORT=8443
SERVER_SSL_ENABLED=false
# 브라우저가 접속하는 URL과 정확히 일치해야 함
REACT_APP_API_BASE_URL=http://localhost:8080
```

기동:

```bash
cd docker-compose-build
docker compose -f docker-compose.yml up -d app
```

헬스체크:

```bash
curl http://localhost:8080/api/config
# {"apiBaseUrl":"http://localhost:8080", ...}  <- 이 값이 위 REACT_APP_API_BASE_URL과 일치해야 정상
```

### 8-3. 빠른 점검 체크리스트

| 증상 | 점검 |
|---|---|
| 로그인 화면이 표시되지 않음 | `docker ps` 로 `testcasecraft` 컨테이너 healthy 상태 확인 |
| 회원가입에서 `Failed to fetch` | `/api/config` 응답의 `apiBaseUrl` 이 브라우저 접속 URL과 일치하는지 확인 |
| `⚠ JIRA` 배지 | 정상 — JIRA 설정 안 했으면 항상 노출. 사용하지 않으면 무시 |
| 헤더에 + 추가 버튼이 안 보임 | 프로젝트 권한이 VIEWER인지 확인 (PM/편집자만 추가 가능) |
| 드래그가 동작 안 함 | VIEWER 권한, 또는 외부 화면(테스트 플랜 등)에서는 선택 모드라 비활성. 우클릭 메뉴/순서 편집 모드 사용 가능 |

## 17. 시스템 관리자 설정 (ADMIN 전용)

> ⚠️ 본 섹션은 `role=ADMIN` 사용자에게만 노출되는 메뉴입니다. 시스템 기본 admin 계정의 초기 비밀번호는 코드에서 `admin / admin123`으로 정의되어 있으며, 운영 환경 배포 직후 반드시 변경하세요.

### 17-1. 관리자 메뉴 진입

관리자 계정으로 로그인하면 헤더에 **[대시보드] / [관리 메뉴 ▾] / [프로젝트 선택]** 이 추가로 노출됩니다.

![관리 메뉴 드롭다운](images/78_admin_menu_dropdown.png)

드롭다운 항목:
- **조직 관리** (`/organizations`)
- **사용자 관리** (`/users`)
- **메일 설정** (`/mail-settings`)
- **LLM 설정** (`/llm-config`)
- **스케줄러 관리** (`/scheduler`)

### 17-2. 전체 대시보드

헤더 좌측 **[대시보드]** 는 전 조직/전 프로젝트를 가로지르는 운영 통계 화면입니다.

![전체 대시보드](images/80_global_dashboard.png)

### 17-3. 조직 관리

`/organizations` — 조직(Organization) 단위로 프로젝트를 그룹핑합니다.

![조직 관리](images/81_organizations.png)

- 조직 신규 생성 / 수정 / 멤버 초대
- 조직 내 그룹(Group) 단위로 권한 매트릭스 관리
- 클릭 시 조직 상세(`/organizations/{id}`)로 진입

### 17-4. 사용자 관리

`/users` — 시스템의 모든 사용자를 조회·관리합니다.

![사용자 관리](images/82_users.png)

- 통계 카드: 전체 / 활성 / 비활성 / 최근 가입
- 검색: 이름·사용자명·이메일
- 필터: 역할(ADMIN/PM/테스터/일반 사용자) · 상태(활성/비활성)
- 행 우측 액션: **👁 상세 보기 · ⋮ 비밀번호 재설정 / 역할 변경 / 활성·비활성 토글**
- 이메일 인증 상태 배지 (미인증/완료)
- 우상단 **🔄 새로고침 / ⬇ 다운로드 / 초기화** 버튼

### 17-5. 메일 설정

`/mail-settings` — SMTP / 발신자 / 템플릿 설정.

![메일 설정](images/83_mail_settings.png)

- SMTP 호스트·포트·계정
- 발신자 이메일 / 표시 이름
- 이메일 인증·비밀번호 재설정·실행 결과 알림 등 시스템 메일 템플릿
- **테스트 메일 발송** 으로 설정 검증

### 17-6. LLM 설정

`/llm-config` — RAG 챗·AI 메타 자동 생성 등에 사용할 LLM 프로바이더 설정.

![LLM 설정](images/84_llm_config.png)

- 프로바이더: OpenAI / Anthropic / Azure OpenAI / 로컬 모델 등
- 모델 명, API Key, 베이스 URL
- 프롬프트 템플릿(시스템 프롬프트 · 예시) 관리
- 사용량 한도와 비용 추적 옵션

### 17-7. 스케줄러 관리

`/scheduler` — 주기적 백그라운드 작업의 켜고/끄기·주기 변경.

![스케줄러 관리](images/85_scheduler.png)

- 등록된 잡 목록 (예: JIRA 동기화, RAG 인덱스 재구축, 만료 토큰 정리)
- 각 잡의 **Cron 표현식**, 마지막 실행 시각/결과
- **수동 실행** 버튼, 일시정지/재개

### 17-8. 번역 관리 (i18n)

`/translation-management` — UI 다국어 키와 번역을 동적으로 편집합니다 (헤더 메뉴에는 기본 비노출 — URL 직접 접근).

![번역 관리](images/86_translation.png)

- 언어별 번역 키/값 편집
- 누락된 키 통계
- 카테고리별 그룹화
- 추가 언어 등록 가능

> 본 화면은 i18n 키가 많아 일반 사용자에게는 보이지 않습니다. 한국어/영어 외 추가 언어를 도입하거나 라벨을 수정할 때만 사용합니다.

### 17-9. 프로젝트 단위 설정 (PM)

프로젝트 카드의 우상단 **⋮ (점 3개)** 메뉴 또는 프로젝트 진입 후 헤더의 메뉴로 접근합니다. (`PROJECT_MANAGER` 권한)

- 프로젝트 이름 / 코드 / 설명 변경
- 멤버 초대 / 역할 부여 (PM / LEAD_DEVELOPER / TESTER / VIEWER)
- 시스템 기본 폴더 / 우선순위 / 태그 컨벤션
- 프로젝트 삭제 (모든 케이스·결과 함께 삭제 — 되돌릴 수 없음)

> 신규 프로젝트는 생성한 사용자가 자동으로 **PROJECT_MANAGER** 가 됩니다. VIEWER 또는 TESTER 권한 사용자는 이 메뉴가 보이지 않거나 일부만 활성화됩니다.
