# SBTM 운영 노트

## 1. 세션 운영 원칙
- 세션은 기본적으로 60~90분의 집중 테스트 블록으로 운영한다.
- 인터럽션(회의, 긴급 대응 등)은 세션 시간에서 제외한다.
- 세션 종료 즉시 리포트를 작성하고 디브리프를 진행한다.
- 리드 승인 전까지 세션 상태는 완료가 아니다.

## 2. 역할과 책임
- 테스터
  - 차터 기반으로 테스트 수행
  - 세션 리포트 작성 및 증적 정리
  - 버그/이슈/질문 분리 기록
- 테스트 리드
  - 디브리프 체크리스트 검토
  - 승인 또는 보완 요청
  - 다음 세션 필요성 판단

## 3. 리포트 작성 규칙
- 차터
  - 실제 수행한 테스트의 중심 미션을 반영해야 한다.
  - 재사용 차터면 이전 세션과의 관계를 명시한다.
- Basic Data
  - 참여자, 환경, 버전, 태그를 누락 없이 입력한다.
- 시간 기록
  - `Test Execution + Bug Investigation + Setup/Admin = 100%`
  - 대략 5% 단위로 현실적으로 기록한다.
- 노트
  - 무엇을 했는지(활동), 어디를 봤는지(커버리지), 어떻게 판단했는지(오라클)를 반드시 포함한다.
- Bugs & Issues
  - 제품 결함(BUG)과 테스트 방해 요소(TEST_IMPEDIMENT), 질문(QUESTION)을 분리한다.

## 4. 디브리프 진행 순서(권장 10~15분)
1. 차터와 실제 활동 일치 여부 확인
2. 시간 비율과 중단 시간 처리 확인
3. 커버리지/오라클/활동 설명 가능성 확인
4. 버그/이슈/질문 누락 여부 확인
5. 다음 세션 필요성 및 차터 후보 도출
6. 승인 또는 보완 요청 결정

## 5. 품질 체크(빠른 검증)
- 이 리포트만 읽고 3개월 뒤에 테스트 상황을 재구성할 수 있는가?
- 왜 이 테스트를 했는지와 무엇을 발견했는지가 분명한가?
- 세션 결과가 다음 실행(수정, 재검증, 추가탐색)으로 연결되는가?

## 6. 안티패턴
- 차터와 무관한 활동이 대부분인데 차터를 수정하지 않음
- 버그 조사 시간을 Test Execution에 섞어 기록함
- "이슈 없음"으로 적었지만 실제로는 질문/혼선/장애물이 있었음
- 스크린샷/로그를 남겼지만 경로를 리포트에 연결하지 않음

## 7. 회고/개선 운영
- 주 1회 30분 회고로 템플릿 부담도와 누락 패턴을 점검한다.
- 자주 누락되는 항목은 템플릿의 필수 항목으로 승격한다.
- 불필요한 항목은 과감히 선택 항목으로 전환한다.

## 8. 권장 KPI
- 세션 승인 리드타임(제출~승인)
- 세션당 버그/이슈 발견 수
- 보완요청률(리포트 재작성 비율)
- 세션 노트 재현성(동료 재해석 성공률)
