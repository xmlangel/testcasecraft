# 릴리즈 절차 및 방법 가이드

이 가이드는 로컬에서 릴리즈를 준비하고 GitHub를 통해 배포를 완료하는 전체 프로세스를 설명합니다.

## 1. 개요
릴리즈 프로세스는 **로컬 준비(수동)**와 **GitHub Actions 배포(자동)**의 두 단계로 나뉩니다.

---

## 2. [1단계] 로컬 준비 (수동 작업)

### ① 버전 업데이트 및 로컬 빌드 (**필수**)
현재 애플리케이션의 버전을 업데이트하고, 변경된 버전이 `package-lock.json` 등에 반영되도록 로컬 빌드를 수행합니다. **태그 생성 전 이 과정을 거쳐야만 배포본과 소스 코드의 메타데이터가 일치하게 됩니다.**
```bash
# 1. 버전 끝자리(Patch) 자동 증가 및 관련 파일 동기화
./gradlew incrementVersion

# 2. 로컬 빌드 수행 (package-lock.json 업데이트 및 검증)
./gradlew build -x test
```
> [!IMPORTANT]
> `./gradlew build`를 실행해야 `src/main/frontend/package-lock.json` 등 빌드 시점에 업데이트되는 파일들이 최신 버전으로 갱신됩니다. **반드시 이 빌드 작업을 마친 후 `git add` 및 커밋을 진행해야 합니다.**

### ② 릴리즈 노트 작성
`docs/release_note/` 디렉토리에 이번 버전에 대한 설명 파일을 생성합니다.
- **한국어**: `RELEASE_NOTE_X.X.X_KO.md`
- **영어**: `RELEASE_NOTE_X.X.X_EN.md`
- *예시: 1.0.35 버전인 경우 `RELEASE_NOTE_1.0.35_KO.md`*

### ③ 빌드 결과물 커밋 및 태그 생성
위 단계에서 수행한 빌드 결과물(`package-lock.json` 등)을 포함하여 커밋하고 태그를 생성합니다.
```bash
# 변경된 모든 파일(버전 정보, package-lock.json 등) 스테이징
git add .
git commit -m "chore: release version 1.0.36"

# 태그 생성 (v버전-대상)
# 빌드 시점에 해당 태그의 존재 유무를 확인하므로 이 시점에 생성하는 것이 표준입니다.
git tag v1.0.36-app
```

### ④ 태그 푸시
로컬의 태그를 GitHub로 푸시하면 배포가 시작됩니다.
```bash
git push origin v1.0.36-all
```

> [!TIP]
> **스크립트 개선 사항 (v1.0.36+)**: 
> 이제 `build-and-push-multiplatform.sh` 등 모든 빌드 스크립트는 실행 시 현재 Git 태그를 자동으로 감지하여 버전으로 사용합니다. 만약 `--version` 인자로 지정한 태그가 로컬에 존재하지 않으면 빌드가 중단되도록 안전 장치가 추가되었습니다.

---

## 3. [2단계] GitHub 배포 (자동화)

태그가 푸시되면 GitHub Actions가 다음 작업을 수행합니다.

1. **Docker 빌드**: 멀티 플랫폼(ARM64/AMD64) 이미지를 빌드합니다.
2. **Docker Hub 푸시**: 빌드된 이미지를 Docker Hub로 전송합니다.
3. **GitHub Release 생성**:
   - 로컬에서 작성한 `docs/release_note/`의 파일을 읽어옵니다.
   - 상단(한국어), 하단(영어) 순으로 내용을 구성하여 GitHub 상에 릴리즈를 생성합니다.

---

## 4. 모니터링 및 결과 확인
- **빌드 상태**: GitHub Repository의 **Actions** 탭에서 로그 확인
- **결과 확인**:
  - **Docker Hub**: `xmlangel/testcasecraft` 이미지 확인
  - **GitHub Release**: 저장소 우측의 **Releases** 메뉴에서 생성된 결과 확인
