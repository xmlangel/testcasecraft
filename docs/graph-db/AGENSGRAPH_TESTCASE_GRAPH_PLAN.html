0. 요약

testcasecraft에 그래프 데이터 모델을 도입한다. 테스트 케이스·실행·결과·오류 사이의 관계를 그래프로 시각화하고, 테스트 케이스 자체도 그래프 기반으로 만들거나 전환할 수 있게 한다. 기존 관계형(1.0.x) 자산은 손대지 않고, 그래프는 그 위에 얹는 추가 기능으로 간다.

항목결정근거
그래프 엔진AgensGraph 2.16.9 별도 인스턴스PG16 기반 → 메인(pgvector/pg18)과 동거 불가
토폴로지메인 DB=SSOT, AgensGraph=그래프 스토어관계형 자산 보존 + 그래프 이점
프런트Cytoscape.js (react-cytoscapejs)대규모 그래프·레이아웃·필터, MUI 병존
버전1.1.1상향 호환(1.0.x→1.1.1 ✅), 하향 비호환(1.1.1→1.0.x ❌)
전환시나리오 선택식 (기본 / 그래프 / 기본→그래프)케이스 단위 표현 선택

1. 두 가지 "그래프" 개념 (혼동 방지)

1-A. 관계 그래프 (시각화용)

프로젝트→폴더→케이스→계획→실행→결과→오류 유형으로 이어지는 관계망. 같은 오류가 어디까지 반복되는지 눈으로 확인하는 용도다. (최초 요청)

1-B. 그래프 테스트 케이스 (저작용)

테스트 케이스 내용을 선형 스텝 대신 노드(스텝/전제/분기/상태)와 엣지(흐름/전이/의존)로 모델링한다. 분기·상태전이는 이쪽이 표현하기 자연스럽다. (신규 요청)

둘 다 같은 AgensGraph 인스턴스 안에 들어가되, 라벨 집합을 다르게 해서 공존시킨다.

2. 현행 분석 (As-Is)

데이터베이스

메인: pgvector/pgvector:pg18DB: testcase_management포트 5434:5432FlywayJPA/Hibernatepostgresql 42.7.11

핵심 엔티티 (그래프 원천)

엔티티그래프 관점 핵심 필드
TestCaseid, displayId, name, type, priority, parentId(폴더계층), steps, expected/pre/postCondition, tags
TestPlanid, name, testCaseIds
TestExecution / TestResultstatus / result(PASS·FAIL·BLOCKED·NOT_RUN), testCaseId, jiraIssueKey
TestSessionBugtitle, severity, jiraIssueKey
JunitTestCasename, className, status, failureType, failureMessage, stackTrace

프런트 / 제약

  • Vite + React 18 + MUI 7 + recharts + axios. 그래프 라이브러리 없음(신규).
  • @tanstack/react-query가 없으므로 그래프 데이터 페칭도 기존 axios + ProjectContext 패턴을 따른다.
  • AgensGraph 2.16.9는 PG16 포크라서 메인(pg18)과 같은 인스턴스에 합칠 수 없다. 별도 컨테이너로 간다.

3. 목표 아키텍처 (To-Be)

메인 DB(SSOT) — AgensGraph 간 양방향 프로젝션, 그래프는 Cytoscape로 렌더

메인 DB가 관계형 원본(SSOT)을 갖고, AgensGraph는 시각화와 그래프 TC 저작을 맡는다. 둘 사이는 양방향 프로젝션으로 잇는다.

4. 그래프 데이터 모델

CREATE GRAPH tc_graph; 후 라벨 정의 (VLABEL=정점, ELABEL=간선).

4-A. 관계 그래프 (시각화)

정점: ProjectFolderTestCaseTestPlanTestExecutionTestResultBugJunitCaseFailureTypeUserJiraIssue

FailureType을 허브로 두는 이유: 같은 근본원인(failureType/stackTrace 시그니처)이 어느 케이스·폴더·실행까지 퍼져 있는지가 그래프에서 바로 보인다.

4-B. 그래프 테스트 케이스 (저작)

정점: GraphTestCaseStepNodePreconditionDecisionStateExpected

기본(선형) TC를 전환하면 스텝 N개가 StepNode N개와 NEXT 간선 N-1개로 매핑된다.

5. 표현 이중화 & 관계형 역투영

요구: "그래프로 만든 테스트 케이스를 관계형 DB에서도 볼 수 있게."

5-A. TestCase 표현 모드 (additive 컬럼)

ALTER TABLE testcase
  ADD COLUMN representation_mode VARCHAR(10) NOT NULL DEFAULT 'BASIC', -- BASIC|GRAPH|HYBRID
  ADD COLUMN graph_vertex_id    VARCHAR(64),  -- AgensGraph 정점 참조
  ADD COLUMN graph_synced_at    TIMESTAMP;
모드SSOT관계형 스텝그래프
BASIC관계형원본(편집)없음
GRAPH그래프프로젝션(read-only)원본(편집)
HYBRID그래프프로젝션(대표경로)원본+분기

5-B. 역투영 (Graph → Relational)

그래프 TC를 testcasesteps로 평탄화하여 기존 스프레드시트/폼(1.0.x UI 포함)에서 조회 가능하게:

  1. STARTS_AT→NEXT 따라 주 경로를 위상정렬 추출.
  2. 분기(Decision) 있으면 대표 경로 선형화 + 주석 → HYBRID.
  3. 각 StepNode → TestStep 행 기록, graph_synced_at 갱신 (미러는 read-only).
이렇게 하면 그래프 TC도 항상 관계형 표에서 조회할 수 있다. 분기 구조 전체는 그래프 뷰에서 보고, 선형 요약은 관계형 쪽에 늘 남는다.

5-C. 편집 충돌 방지

편집은 SSOT 쪽에서만 허용하고, 반대편은 재생성으로만 갱신한다. 양쪽을 따로 고치는 일이 생기지 않는다. UI에는 "그래프 모드 — 관계형 읽기 전용" 배지를 띄운다.

6. 시나리오 선택식 저작·전환

A · 기본 TC

지금과 완전히 같은 동작이다(BASIC). AgensGraph를 거치지 않으므로 회귀 걱정이 없다.

B · 그래프 TC (신규)

Cytoscape 편집기에서 노드를 배치해 저장하면, 메타 행 생성과 AgensGraph 서브그래프 MERGE, 관계형 역투영이 한 번에 이뤄진다.

C · 기본 → 그래프 전환 (마이그레이션)

전환은 비파괴적이다(원본은 미러로 남는다). 폴더/프로젝트 단위 일괄 전환은 P5에서 붙이고, 롤백은 미러를 SSOT로 되돌리는 것으로 처리한다.

7. 동기화 / ETL

7-A. 관계 그래프 (파생)

  • 초기 풀 적재: SELECT → 멱등 MERGE.
  • 증분: @Scheduled(예 5분) + 도메인 이벤트 훅(event 패키지) + 관리자 수동(POST /api/graph/sync).
  • 멱등: MERGE (n:Label {id}) SET ….

7-B. 그래프 TC (역투영)

저장/편집 → 즉시 관계형 미러 재생성(§5-B). 실패 시 graph_synced_at 미갱신으로 감지 + 재시도 큐.

정합성

관계 그래프=최종 일관성(재적재 가능) · 그래프 TC 미러=쓰기 직후 일관성 목표.

8. 백엔드 구현

8-A. 두 번째 DataSource

AgensGraph=표준 PG 프로토콜 → org.postgresql 드라이버 그대로. GraphDataSourceConfig + 전용 graphJdbcTemplate. 세션마다 SET graph_path=tc_graph;.

8-B. Cypher 실행 & JSON

SQL 문맥에서 MATCH…RETURN 직접 실행 → vertex/edge → GraphResponse{nodes[],edges[]}. 파라미터화 + 라벨 화이트리스트(인젝션 방지).

8-C. REST 엔드포인트 /api/graph/*

메서드경로설명
GET/project/{id}/structure구조 그래프
GET/failures?projectId=&from=&to=오류 클러스터
GET/testcase/{id}/neighborhood?depth=케이스 이웃
GET/testcase/{id}/steps그래프 TC 스텝
POST/testcase/{id}/convert기본→그래프 (C)
POST/testcase그래프 TC 신규 (B)
PUT/testcase/{id}/steps그래프 TC 편집
POST/testcase/{id}/revert그래프→기본 역전환
POST/sync관계 그래프 재동기(관리자)

모든 엔드포인트에 @PreAuthorize와 프로젝트 스코프 검증을 건다. AgensGraph는 Cypher에도 RLS가 걸리므로 DB 레벨에서 한 번 더 막을 수 있다. 통합테스트는 Testcontainers로 AgensGraph를 띄워서 돌린다.

9. 프런트엔드 (Cytoscape.js)

9-A. 의존성

"cytoscape":"^3.30.0", "react-cytoscapejs":"^2.0.0",
"cytoscape-fcose":"^2.2.0",  // force-directed
"cytoscape-dagre":"^2.5.0"    // DAG/흐름(그래프 TC)

9-B. 화면/라우트

10. 버전 & 호환성 (1.0.x → 1.1.1)

방향지원설명
1.0.x 데이터 → 1.1.1✅ 상향 호환기존 케이스 BASIC 기본값으로 그대로 동작. 추가 컬럼/그래프는 옵셔널.
1.1.1 → 1.0.x❌ 하향 비호환GRAPH/HYBRID 케이스 완전 표현 불가. 다운그레이드 미지원.

상향 호환 보장 설계

  1. Additive-only 마이그레이션(Flyway): 신규 컬럼 nullable/DEFAULT, 기존 불변.
  2. API 버저닝: /api/v1 계약 불변, 그래프는 신규 /api/graph만.
  3. 관계형 미러 보장(§5-B): 그래프 미지원 소비자도 최소 선형 표현 확보.
  4. Feature flag features.graph.enabled: 미활성 시 1.0.x와 동일.
릴리즈 노트 명시: "1.1.1에서 생성한 그래프 테스트 케이스는 1.0.x 다운그레이드 시 완전 복원 불가"(추가 컬럼이라 크래시는 없으나 미러만 조회).

incrementVersion Gradle 태스크로 app 1.0.x→1.1.1 범프. 이미지 xmlangel/testcasecraft:1.1.1 + skaiworldwide/agensgraph:v2.16.0-bookworm.

11. 배포 (docker-compose)

  agensgraph:
    image: skaiworldwide/agensgraph:v2.16.0-bookworm
    container_name: testcasecraft-agensgraph
    environment:
      POSTGRES_DB: tc_graph_db
      POSTGRES_USER: graph_user
      POSTGRES_PASSWORD: ${GRAPH_DB_PASSWORD}
    ports: ["5437:5432"]          # 메인(5434)과 분리
    volumes:
      - ./data/agensgraph:/var/lib/postgresql/data
      - ./init-scripts/graph:/docker-entrypoint-initdb.d  # CREATE GRAPH+라벨
    healthcheck:
      test: ["CMD-SHELL","pg_isready -U graph_user -d tc_graph_db || exit 1"]

app env: GRAPH_DB_URL=jdbc:postgresql://agensgraph:5432/tc_graph_db, FEATURES_GRAPH_ENABLED=true.

이미지 태그 2.16.9 존재 여부는 P0 검증 — 없으면 소스 빌드(github.com/skaiworldwide-oss/agensgraph) 또는 최신 2.16 태그.

12. 단계별 로드맵

P0
PoC/검증 — AgensGraph 2.16.9 기동, 이미지 태그·PG16.9 base 확인, 샘플 그래프+Cypher 검증. 3~5일 · 선행 게이트
P1
데이터 모델 + 초기 ETL — §4 라벨·인덱스, 관계 그래프 풀 적재 배치. 1~2주
P2
백엔드 그래프 API — 두 번째 DataSource, Cypher→JSON, 조회 API. 1~2주
P3
표현 이중화 & 전환 — representation_mode 마이그레이션, 변환기(B/C/역투영). 1.5~2주
P4
프런트 (Cytoscape) — 구조·오류·이웃 뷰 + 그래프 TC 편집기, i18n 한/영. 2~3주
P5
증분 동기화 & 일괄 전환 — @Scheduled/이벤트, 폴더/프로젝트 bulk 전환, 관리자 UI. 1주
P6
마감 — 1.1.1 범프, 매뉴얼(한/영), E2E, 릴리즈 노트. 1주

1인 기준 8~11주 정도로 본다(병렬로 나누면 단축). P0가 통과돼야 나머지를 시작한다.

13. 리스크 & 완화

리스크완화
PG 버전 차이(16 vs 18)별도 인스턴스(확정). 메인 pg18 유지 — 마이그레이션 아님.
2.16.9 태그 부재 가능P0 검증. 소스 빌드/최신 2.16 태그 대안.
동기화 지연·정합성관계 그래프=최종 일관성+재적재. 미러=쓰기직후+재시도.
그래프 규모 폭증프로젝트 스코프 + depth/페이지네이션 + 뷰포트 컬링.
이중 편집 충돌SSOT 단일화, 반대편 read-only + regenerate.
Cypher 인젝션파라미터화 + 화이트리스트 + RLS 이중 방어.
하향 비호환 오해릴리즈 노트/매뉴얼 명시, feature flag 점진 활성.
react-query 오도입axios+ProjectContext 강제.
